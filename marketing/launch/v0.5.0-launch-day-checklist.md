# v0.5.0 launch-day checklist

A condensed, today-only version of `docs/RELEASING.md` — strip it down to what actually needs doing on the day of the announcement. Tick the boxes as you go.

## Before tagging (≤ 1 hour)

- [ ] `git status` is clean on `main`. No uncommitted work, no untracked files outside `marketing/`.
- [ ] `dotnet build -c Release -warnaserror` is green locally.
- [ ] `dotnet test -c Release` is green locally — all 75+ cases pass.
- [ ] `dotnet pack -c Release --output ./artifacts` produces one `.nupkg` per `src/` project (≥ 15 packages).
- [ ] Quick smoke test: `dotnet run --project samples/Mvc.Quickstart`, hit `/auth/sign-in`, send yourself a magic link, click it. Should land back on `/`.
- [ ] `marketing/launch/release-notes-v0.5.0.md` reflects what's actually shipping. No "TBD" or stale claims.
- [ ] `NUGET_API_KEY` GitHub secret is set (verify under Settings → Secrets → Actions).
- [ ] Repo is public (or scheduled to flip public the moment the tag is pushed).

## Cut the release

- [ ] `git tag -a v0.5.0 -m "TapInAuth v0.5.0 — first public release"`
- [ ] `git push origin v0.5.0`
- [ ] Watch the Release workflow finish on the Actions tab. ~5 min.
- [ ] Verify on nuget.org that at least the metapackage and `TapInAuth.AspNetCore` are indexed.
- [ ] Open the auto-generated GitHub Release; replace the body with the contents of `release-notes-v0.5.0.md` if the auto-notes are rough.

## Post the launch (within 1 hour of release)

- [ ] **Twitter/X**: paste `marketing/launch/twitter-thread.md` as a thread. One tweet per `**N/7**` block.
- [ ] **LinkedIn**: paste `marketing/launch/linkedin-post.md` as a single post.
- [ ] **Blog**: publish `marketing/launch/blog-announcement.md` on dev.to (and crosspost to your personal blog if you have one).

## Late morning push (~T+2 hr)

- [ ] **Hacker News**: submit using `marketing/launch/hacker-news.md`. Title, URL, then post the body as the first comment yourself.
- [ ] **/r/dotnet**: post `marketing/launch/reddit-r-dotnet.md` (title + body).

## Active triage (rest of the day)

- [ ] Watch the GitHub Issues tab — bots first, then humans. Triage anything that looks like a security report into a private Security Advisory immediately.
- [ ] Reply to every comment thread within ~1 hr. Speed of response on day 1 sets the tone for the project.
- [ ] If a HN thread takes off, expect a P95 traffic spike on `tapinauth.io` for ~3 hours. If you're hosting docs there, confirm it's CDN-fronted.

## If something goes wrong

| Symptom                                          | First move                                                      |
| ------------------------------------------------ | --------------------------------------------------------------- |
| Release workflow fails on build/test             | Fix on main, delete the tag, retag                              |
| Workflow fails on NuGet push (rate limit / dup)  | Re-run via `workflow_dispatch` from the Actions tab             |
| Package indexed on nuget.org with a bug          | Unlist on nuget.org, push a patch (`v0.5.1`) ASAP               |
| Comment thread on HN/Reddit going hostile        | Reply once with technical substance, then disengage             |
| Security report in DMs / email                   | Acknowledge in 1 hr, work it under coordinated disclosure       |

## What "good" looks like at end of day

- ≥ 1 NuGet install on `TapInAuth.AspNetCore` from someone you don't know.
- ≥ 1 GitHub star from someone you don't know.
- ≥ 1 substantive comment on HN OR /r/dotnet.
- Zero open issues with `bug` label that you haven't acknowledged.

Anything beyond that is gravy. Anything short of that, take notes — what would the announcement have to say differently for it to break through?
